# Meta Selector: простое объяснение на локальном коде

Этот документ описывает именно текущую локальную реализацию из:

- [`src/domain/meta_selector.py`](/mnt/w/_python/MT/src/domain/meta_selector.py)
- [`src/domain/meta_selector_ml.py`](/mnt/w/_python/MT/src/domain/meta_selector_ml.py)
- [`src/domain/meta_selector_outputs.py`](/mnt/w/_python/MT/src/domain/meta_selector_outputs.py)
- [`src/bokeh_app/main.py`](/mnt/w/_python/MT/src/bokeh_app/main.py)

## 1. Что Meta Selector делает в одном предложении

Meta Selector не ищет новые параметры с нуля.

Он берет уже сохраненную историю WFA-оптимизаций, учится на старых окнах, потом для каждого OOS-окна пытается предсказать, какой набор параметров сработает лучше.

## 2. Главная идея очень простыми словами

Есть WFA-история.

В каждом WFA-окне было протестировано много наборов параметров:

- `lookback_bars`
- `entry_z`
- `exit_z`
- `stop_z`
- `bollinger_k`

Для каждого такого набора уже известны результаты:

- train-метрики
- test-метрики

Meta Selector делает вот что:

1. Берет старые WFA-строки до выбранной даты `OOS Start`.
2. Обучает ML-модель предсказывать целевую метрику.
3. На более новых WFA-строках после `OOS Start` считает прогноз `predicted_target`.
4. Строит две разные таблицы:

- `Meta Robustness Grid`: агрегированный глобальный рейтинг наборов параметров
- `Selected OOS Folds`: финальный выбор по одному набору на каждое OOS-окно

Это две разные сущности. Именно отсюда и идет путаница.

## 3. Схема потока данных

```text
WFA optimization history
        |
        v
фильтр по последнему wfa_run_id
        |
        v
разделение по дате OOS Start
  pre-OOS rows -> обучение модели
  post-OOS rows -> прогнозы модели
        |
        +------------------------------+
        |                              |
        v                              v
Meta Robustness Grid          Selected OOS Folds
group by parameter set        choose 1 best row per OOS window
global ranking                final schedule "window by window"
        |                              |
        +---------------+--------------+
                        |
                        v
             Meta Stitched Equity
     re-backtest выбранных OOS-окон по очереди
```

## 4. Очень простой игрушечный пример

Представим, что после `OOS Start` у нас есть только 2 OOS-окна:

- окно `W1`
- окно `W2`

И в каждом окне модель оценивает только 2 набора параметров:

- набор `A`
- набор `B`

### 4.1. Что видит Meta Selector

```text
W1:
  A -> predicted_target = 0.80
  B -> predicted_target = 0.55

W2:
  A -> predicted_target = 0.20
  B -> predicted_target = 0.70
```

### 4.2. Что получится в `Selected OOS Folds`

По каждому окну выбирается лучший прогноз:

```text
W1 -> выбрать A
W2 -> выбрать B
```

Итог:

```text
Selected OOS Folds
  W1 -> A
  W2 -> B
```

То есть нижняя таблица не обязана состоять из одного и того же набора параметров.

Она может быть "лоскутной":

- в одном окне победил `A`
- в другом победил `B`

### 4.3. Что получится в `Meta Robustness Grid`

Здесь строки агрегируются по самому набору параметров.

То есть отдельно считается статистика для `A` и отдельно для `B`.

Для примера:

```text
A:
  predictions = [0.80, 0.20]
  Pred Mean = 0.50
  Pred Std  = 0.30

B:
  predictions = [0.55, 0.70]
  Pred Mean = 0.625
  Pred Std  = 0.075
```

Если у `B` средний прогноз выше и разброс меньше, `B` может быть выше в `Meta Robustness Grid`.

Но это не означает, что `Selected OOS Folds` должны стать "везде только B".

Почему:

- `Meta Robustness Grid` отвечает на вопрос: "какой набор в среднем выглядит устойчивым"
- `Selected OOS Folds` отвечает на вопрос: "какой набор выбран в каждом конкретном окне"

## 5. Что такое `Meta Robustness Grid`

Это глобальный рейтинг наборов параметров, агрегированный по OOS-части истории.

Группировка идет по полям:

- `lookback_bars`
- `entry_z`
- `exit_z`
- `stop_enabled`
- `stop_z_value`
- `bollinger_k`

То есть одна строка в `Meta Robustness Grid` = один конкретный набор параметров.

### 5.1. Расшифровка колонок `Meta Robustness Grid`

#### `Rank`

Позиция строки после сортировки.

Чем меньше число, тем выше набор в текущем рейтинге.

#### `Stability`

Это `stability_score`.

В коде он считается так:

```text
stability_score =
    predicted_mean
    - 0.50 * predicted_std
    + 0.05 * train_pnl_to_maxdd_mean
```

Смысл:

- хотим высокий средний прогноз
- хотим меньший разброс прогнозов
- слегка поощряем хорошие train-отношения profit / drawdown

Это эвристика ранжирования, а не "реальные деньги".

#### `Pred Mean`

Это `predicted_mean`.

Среднее значение прогноза модели `predicted_target` для этого набора параметров по всем его OOS-строкам.

Важно:

- это не факт
- это не реальный PnL
- это средняя оценка модели

Если целевая метрика сейчас `test_score_log_trades`, то:

```text
Pred Mean = средний прогноз модели по test_score_log_trades
```

#### `Pred Std`

Это `predicted_std`.

Стандартное отклонение прогнозов модели для этого набора параметров по всем OOS-строкам.

Простой смысл:

- маленький `Pred Std` = модель считает поведение набора более ровным
- большой `Pred Std` = модель считает набор более дерганым от окна к окну

#### `Test Score`

Это `actual_test_score_mean`.

Среднее реальное значение `test_score_log_trades` по OOS-строкам этого набора.

Это уже не прогноз, а фактический результат из WFA history.

#### `Test Net`

Это `actual_test_net_mean`.

Средний фактический `test_net_profit` по OOS-строкам этого набора.

#### `Test DD`

Это `actual_test_maxdd_mean`.

Средняя фактическая `test_max_drawdown` по OOS-строкам этого набора.

#### `Trades`

Это `actual_test_trades_mean`.

Среднее число сделок в test/OOS части для этого набора.

#### `Train Score`

Это `train_score_mean`.

Средний `train_score_log_trades` по train-частям WFA для этого набора.

#### `Train Net`

Это `train_net_mean`.

Средний `train_net_profit` по train-частям WFA для этого набора.

#### `Rows`

Количество OOS-строк, которые попали в эту агрегированную строку.

Проще:

```text
сколько раз этот exact-набор параметров вообще встретился в OOS-истории
```

#### `Folds`

Сколько разных fold/window содержали этот набор параметров.

Практический смысл почти такой же:

```text
в скольких OOS-окнах этот набор вообще присутствовал
```

#### `Lookback`

Это `lookback_bars`.

Размер окна lookback в барах.

#### `Entry`

Это `entry_z`.

Z-score входа.

#### `Exit`

Это `exit_z`.

Z-score выхода.

#### `Stop`

Это `stop_z_label`.

Либо:

- `disabled`
- либо конкретное значение stop по z-score

#### `Boll`

Это `bollinger_k`.

Коэффициент Bollinger.

## 6. Что такое `Selected OOS Folds`

Это финальная "раскладка по окнам".

Одна строка = один выбранный набор параметров для одного конкретного OOS-окна.

Логика выбора в коде такая:

1. Для каждой OOS-строки считается `predicted_target`.
2. Внутри каждого окна строки сортируются по:

- `predicted_target` по убыванию
- потом `train_score_log_trades`
- потом `train_net_profit`
- потом `train_pnl_to_maxdd`
- потом `trial_id`

3. Берется первая строка.

Именно это и есть "выбранный победитель" данного окна.

### 6.1. Важный вывод

`Selected OOS Folds` не является детализацией выбранной строки из `Meta Robustness Grid`.

Это отдельный финальный результат:

```text
по одному лучшему кандидату на каждое OOS-окно
```

### 6.2. Расшифровка колонок `Selected OOS Folds`

#### `Fold`

Номер WFA-окна.

#### `Test From`, `Test To`

Начало и конец OOS/test-окна.

#### `Pred`

Это `predicted_target`.

Прогноз модели именно для этой конкретной строки-окна.

Если целевая метрика = `test_score_log_trades`, то:

```text
Pred = прогноз модели, каким будет test_score_log_trades
для этого набора параметров в этом окне
```

#### `Score`

Это `test_score`.

Важно:

в текущем коде сюда кладется сохраненный `test_score_log_trades` из WFA history, а не заново пересчитанный score.

То есть это фактический WFA test-score выбранной строки.

#### `Net`

Это `test_net_profit`.

Здесь уже используется повторный backtest выбранного набора на этом OOS-окне.

#### `Max DD`

Это `test_max_drawdown`.

Тоже берется из повторного backtest выбранной строки на этом окне.

#### `Trades`

Это `test_trades`.

Количество сделок из повторного backtest на этом окне.

#### `Lookback`, `Entry`, `Exit`, `Stop`, `Boll`

Это конкретные параметры, выбранные для данного OOS-окна.

#### `Comm`

Это `test_commission`.

Комиссия из повторного backtest выбранной строки на этом окне.

#### `Costs`

Это `test_total_cost`.

Полная стоимость из повторного backtest выбранной строки на этом окне.

## 7. Что именно рисует `Meta Stitched Equity`

`Meta Stitched Equity` строится не по `Meta Robustness Grid`.

Он строится по `Selected OOS Folds`.

Процесс такой:

1. Выбрали по одной строке на каждое OOS-окно.
2. Для каждой выбранной строки повторно запустили backtest именно на ее OOS-окне.
3. Куски equity последовательно "сшили" друг за другом.

То есть этот график показывает:

```text
что получилось бы, если в каждом OOS-окне брать именно тот набор,
который Meta Selector выбрал для этого окна
```

## 7.1. Почему может казаться, что Meta Selector "берет только 2025 год"

Это самый частый источник путаницы.

Meta Selector использует два разных временных блока:

- `train/pre-OOS history`
- `OOS history`

Если в UI стоит, например:

```text
OOS Start = 2025-08-01
```

то происходит вот что:

```text
2024-... -> 2025-07-31  = используется для обучения модели
2025-08-01 -> дальше    = используется для прогнозов, выбора OOS-folds и stitched equity
```

Поэтому:

- `Meta Stitched Equity` стартует только после `OOS Start`
- `Selected OOS Folds` тоже начинаются только после `OOS Start`
- это не значит, что 2024 проигнорирован
- это значит, что 2024 ушел в train-часть, а не в OOS-визуализацию

Проще:

```text
2024 может участвовать в обучении
и одновременно вообще не появляться на Meta Stitched Equity
```

Это нормальное поведение текущей логики.

## 8. Должна ли `Selected OOS Folds` меняться при клике на строку в `Meta Robustness Grid`

Короткий ответ:

нет, в текущей логике не должна.

Почему:

- `Meta Robustness Grid` = глобальный рейтинг наборов параметров
- `Selected OOS Folds` = финальная покадровая раскладка "одно окно -> один победитель"

Это разные уровни представления.

### 8.1. Что происходит сейчас при клике на строку в `Meta Robustness Grid`

Сейчас клик по строке `Meta Robustness Grid` делает вот что:

1. берет параметры этой строки
2. копирует их в основной tester
3. запускает обычный tester на полном текущем tester-периоде

То есть это действие:

```text
"протестировать этот набор руками в основном tester"
```

А не:

```text
"перестроить Selected OOS Folds под эту строку"
```

### 8.2. Почему сейчас визуально "ничего не меняется"

Потому что:

- нижняя таблица не фильтруется
- `Meta Stitched Equity` не перестраивается
- меняется только состояние основного tester и текст статуса

Если смотреть только на meta-блок, действительно кажется, что клика "нет".

## 9. Было бы логично сделать по-другому?

Да, как UX-улучшение это можно сделать понятнее.

Есть 3 возможных варианта правильного поведения по клику на `Meta Robustness Grid`:

### Вариант A

Ничего не менять в `Selected OOS Folds`, но явно написать в UI:

```text
клик по grid просто копирует параметры в tester
```

Это минимальное и честное поведение.

### Вариант B

Фильтровать нижнюю таблицу и показывать:

```text
как этот конкретный набор параметров выглядел по всем OOS-окнам
```

Это уже будет drill-down по строке grid.

### Вариант C

Перестраивать отдельный альтернативный stitched equity:

```text
"что было бы, если брать один и тот же grid-набор на всех OOS-окнах"
```

Это, возможно, самый наглядный вариант для сравнения:

- текущий meta-выбор по окнам
- один fixed robust-набор на всех окнах

Но это уже другая аналитика, не текущая.

## 10. Практическая интерпретация без путаницы

Самый полезный способ читать экран сейчас такой:

### Смотри на `Meta Robustness Grid`, чтобы понять

- какие наборы в среднем выглядят сильными
- какие наборы более стабильные
- какие наборы меньше дергаются между окнами

### Смотри на `Selected OOS Folds`, чтобы понять

- какой набор реально выбран для каждого OOS-окна
- какую лоскутную последовательность параметров собрал Meta Selector
- из чего складывается `Meta Stitched Equity`

## 11. Короткий словарь

### `OOS`

Out-of-sample.

То есть более "будущая" часть данных, на которой набор уже проверяется, а не подбирается.

### `Fold`

Одно WFA-окно.

### `Row`

Одна строка истории WFA:

```text
одно окно + один конкретный набор параметров + его метрики
```

### `Parameter set`

Конкретная комбинация:

- `lookback_bars`
- `entry_z`
- `exit_z`
- `stop_z`
- `bollinger_k`

### `predicted_target`

Прогноз модели для одной конкретной строки.

### `Pred Mean`

Средний `predicted_target` по всем OOS-строкам данного набора параметров.

### `Pred Std`

Разброс `predicted_target` по всем OOS-строкам данного набора параметров.

### `Stability`

Комбинированный score ранжирования:

- выше средний прогноз
- ниже разброс
- чуть лучше train pnl/maxdd

## 12. Самый короткий итог

Если совсем кратко:

- верхняя таблица = "какие наборы вообще выглядят устойчивыми"
- нижняя таблица = "какой набор выбран в каждом конкретном OOS-окне"
- нижняя таблица сейчас не обязана меняться при клике по верхней
- сейчас клик по верхней таблице просто отправляет этот набор в основной tester
